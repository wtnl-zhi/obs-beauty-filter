# OBS 双端高阶美颜滤镜插件：开发说明（评审稿）

> 实施状态（2026-07-15）：P0 已完成 macOS arm64 本机构建验证。P1 已完成 MediaPipe Face Landmarker 的 Apple Silicon 原生运行时、模型加载、关键点适配、有界后台推理工作器、OBS 双 staging 纹理帧桥及 GPU 关键点驱动的局部 mask，并通过真实模型推理测试；像素级皮肤分割不在当前范围。

## 1. 文档目的

本文定义一款 OBS Studio 视频滤镜插件的首版开发范围，供产品、算法与客户端开发评审。插件用于在本地实时处理摄像头画面，提供自然美颜能力，并支持 Windows 与 Apple Silicon Mac。

本文中的“首版”指可安装、可配置、可在直播中稳定运行的产品版本；本文不会将尚未确认的特效纳入承诺范围。

## 2. 已确认范围

### 2.1 支持平台

| 平台 | 支持范围 | 目标架构 |
| --- | --- | --- |
| Windows | 支持 | x64 |
| macOS | 仅支持 Apple Silicon | arm64 |
| Intel Mac | 不支持 | x86_64 |
| Linux | 不支持（首版） | - |

### 2.2 OBS 集成方式

- 作为 **OBS 视频源滤镜（Video Source Filter）** 出现，可添加到“视频采集设备”等视频源。
- 每个滤镜实例拥有独立的配置、模型会话和运行状态。
- 支持与 OBS 原有滤镜、背景移除等第三方滤镜串联；首版应在产品说明中标注推荐滤镜顺序。
- 不上传摄像头画面，不依赖网络服务；所有推理和渲染在用户本机完成。

### 2.3 首版功能

| 能力 | 首版状态 | 说明 |
| --- | --- | --- |
| 磨皮 | 包含 | 保边缘的局部平滑，避免整幅画面发糊。 |
| 美白/肤色 | 包含 | 对皮肤 mask 内的亮度、饱和度与色相做温和调整。 |
| 红润 | 包含 | 对皮肤区域进行可控的暖色补偿。 |
| 锐化 | 包含 | 在美颜后做轻度细节恢复；可关闭。 |
| 人脸检测与皮肤/人脸遮罩 | 包含 | 用于将效果限制在脸部与皮肤区域。 |
| 多人脸 | 包含 | 首版最多处理 4 张脸；超过时优先处理面积较大的脸。 |
| 强度与开关 | 包含 | 每项能力可单独开关、调节强度。 |
| 预设 | 包含 | 至少提供“自然”“清晰”“直播”三个本地预设。 |
| 瘦脸/大眼 | 暂不包含 | 需要关键点稳定、网格形变与更严格的效果质量控制。 |
| 妆容/口红/美瞳 | 暂不包含 | 需要高精度区域分割与更完整的素材、审美方案。 |
| 祛痘/瑕疵修复 | 暂不包含 | 需要专用修复模型，暂不承诺效果。 |

## 3. 体验与性能目标

### 3.1 性能原则

1. 不能在 OBS 的渲染线程执行模型推理。
2. 模型只处理降采样图像（建议最长边 256–512 px）；最终全分辨率画面由 GPU Shader 合成。
3. 检测与分割以低频更新，渲染帧使用跟踪、平滑后的 mask，避免卡顿和 mask 抖动。
4. 首选稳定和自然；当性能不足时自动降级，而不是阻塞输出。

### 3.2 性能档位

| 档位 | 典型硬件 | 目标能力 | 自动策略 |
| --- | --- | --- | --- |
| 兼容 | Windows 4 核 CPU / 8GB 内存 | 720p30、基础磨皮与肤色 | CPU 推理，低分辨率 mask，隔帧更新。 |
| 推荐 | Windows 6 核+ / 16GB + DX12 GPU；或 M1+ / 16GB | 1080p30、多人脸、全部首版功能 | GPU/系统加速，平滑 mask。 |
| 高性能 | 现代独显 4GB+ 显存；或 M1 Pro/M2+ / 16GB+ | 1080p60、多人脸与高质量 mask | 提升推理输入尺寸和更新频率。 |

上述为开发验收的基准档，而非对所有摄像头、编码器和场景复杂度的绝对保证。实际性能同时受 OBS 场景、编码方式和分辨率影响。

### 3.3 运行时降级

- 优先降低推理分辨率，其次降低检测频率，最后限制同时处理的人脸数。
- 若后台推理滞后，渲染继续使用最近一帧有效 mask；不得等待推理结果。
- 无可用加速后端时切换 CPU 后端，并在 UI 提示“兼容模式”。
- 初始化模型失败时仅禁用 AI 局部能力；不应导致 OBS 崩溃或视频源黑屏。

## 4. 技术架构

```text
OBS 输入视频帧
    │
    ├── 渲染线程：原始纹理 ──> Shader 合成（磨皮/肤色/锐化）──> OBS 输出
    │                                  ▲
    │                                  │ 人脸/皮肤 mask（GPU 纹理）
    │
    └── 后台推理线程：降采样帧 ──> 人脸检测/分割 ──> 跟踪与时序平滑 ──┘
```

### 4.1 模块划分

| 模块 | 职责 |
| --- | --- |
| OBS Filter Adapter | 注册滤镜、创建/销毁实例、读取属性、串联前后滤镜。 |
| Frame Bridge | 从 OBS 图形纹理安全取得最长边 256px 的低分辨率输入；双 staging surface 非阻塞读回，最多每秒提交 10 帧。 |
| Inference Engine | 加载模型、选择执行后端、在单独工作线程异步运行人脸检测与分割；队列仅保留最新帧。 |
| Face Tracker | 关联多帧人脸、平滑位置/尺寸/置信度，防止跳动。 |
| Mask Processor | 基于脸部椭圆与五官保护区生成 GPU 局部 mask；后续增加羽化、扩张/收缩、时序稳定与遮挡处理。 |
| Beauty Renderer | 基于 OBS `gs_effect` / `.effect` Shader 执行局部美颜。 |
| Settings & Presets | 持久化参数、版本迁移、预设加载与保存。 |
| Telemetry/Logging | 记录后端、耗时、降级和错误；首版不采集用户图像。 |

### 4.2 推理后端

| 平台 | 优先后端 | 后备后端 | 说明 |
| --- | --- | --- | --- |
| Windows x64 | MediaPipe CPU / XNNPACK | P0 Shader 回退；后续评估 DirectML | P1 提供原生构建脚本；尚待 Windows x64 实机验证。 |
| macOS arm64 | MediaPipe CPU / XNNPACK | 后续评估 Core ML | 当前已验证原生 CPU 路径；不在渲染线程调用。 |

推理层应对上提供统一接口，业务和渲染层不得直接依赖 Direct3D、Metal 或某一家 GPU API。模型与运行时的许可证必须在接入前单独审查。

#### P1 选型决定（2026-07-15）

P1 优先接入 MediaPipe Face Landmarker：它以视频模式输出人脸关键点，插件据此在 Shader 中生成脸部椭圆 mask，并扣除眼睛、嘴唇等应保护区域。这样可先实现稳定的“仅脸部美颜”，且不在首个 AI 版本就绑定人脸解析模型。macOS arm64 已验证 CPU/XNNPACK 运行时与最小 OpenCV Core/Imgproc 依赖；Windows x64 已提供同一 C API 的原生构建脚本，尚待实机验证。具体模型包、版本、校验值和许可证记录在 `docs/THIRD_PARTY.md`。

### 4.3 渲染算法（首版）

1. 利用人脸/皮肤 mask 限定处理区域。
2. 在线性或适合肤色处理的色彩空间中完成保边缘平滑。
3. 将平滑结果按强度与原图混合；眼睛、眉毛、嘴唇和头发区域应尽量保护。
4. 仅对 mask 内调整亮度、饱和度、色温/色相，限制参数上限以避免过曝、偏色。
5. 以低强度高频细节恢复替代重度锐化，避免噪点和光晕。

## 5. 用户界面与参数

滤镜名称暂定：`高级美颜`。

| 分组 | 参数 | 范围/默认值（建议） |
| --- | --- | --- |
| 总开关 | 启用美颜 | 默认开启 |
| 预设 | 自然 / 清晰 / 直播 / 自定义 | 默认“自然” |
| 磨皮 | 开关与强度 | 默认开启；0–100，默认 35 |
| 磨皮 | 保留细节 | 0–100，默认 70 |
| 肤色 | 提亮 | 默认开启；0–100，默认 15 |
| 肤色 | 红润 | 默认开启；0–100，默认 10 |
| 肤色 | 肤色均匀 | 0–100，默认 20 |
| 细节 | 锐化 | 默认开启；0–100，默认 10 |
| 性能 | 质量模式 | 自动 / 兼容 / 高质量，默认“自动” |
| 人脸区域 | 边缘羽化 | 0–30，默认 12；仅影响关键点驱动的脸部轮廓与五官保护区 |
| 调试 | 显示人脸区域 | 默认关闭，仅开发/诊断使用 |

内置预设使用以下固定值；用户调整任一滑杆后，预设自动显示为“自定义”。设置包含内部版本号；从无预设的早期版本升级时，既有滑杆值会迁移为“自定义”，避免覆盖用户原有效果。

| 预设 | 磨皮 | 细节 | 提亮 | 红润 | 锐化 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 自然 | 35 | 70 | 15 | 10 | 10 |
| 清晰 | 25 | 85 | 10 | 5 | 22 |
| 直播 | 50 | 55 | 25 | 15 | 8 |

质量模式还决定推理采样档：自动为最长边 256px、10fps；兼容为 192px、6fps；高质量为 384px、15fps。三个档位均使用后台最新帧队列，不能等待推理完成。

诊断模式会以蓝红图显示 GPU 人脸区域，用于检查关键点、帧桥和纹理坐标是否一致；人脸轮廓及五官保护区会按双眼连线旋转对齐。诊断默认关闭，且不保存或上传图像。

UI 中应显示当前状态：`MediaPipe 人脸区域已启用`、`兼容模式`、`模型加载中` 或 `模型不可用`。模型不可用或后台推理报错时自动回退为全画面 Shader 美颜；错误日志应包含可执行的建议，而非仅输出错误码。

## 6. 滤镜顺序建议

默认建议：

```text
摄像头源 → 高级美颜 → 背景移除/虚拟背景 → 色彩校正 → 输出
```

原因：美颜应读取完整的人脸与皮肤信息；过早背景移除可能造成头发和边缘 mask 不稳定。若用户使用降噪或锐化类滤镜，建议放在美颜之前或降低其强度，避免叠加产生伪影。

## 7. 安装、签名与发布

### 7.1 Windows

- 发布 x64 安装包，按 OBS 推荐目录结构安装 `bin/64bit` 与 `data` 文件。
- 安装包包含插件 DLL、Shader、模型、语言文件和必要的运行时依赖。
- 启动时探测系统和 GPU 能力；不可用时提供 CPU fallback。

### 7.2 macOS

- 仅发布 arm64 `.plugin` bundle；启动时拒绝 Intel Mac，并说明不支持原因。
- 包含 arm64 原生依赖、Core ML 模型和资源文件。
- 发布前完成 Apple Developer ID 签名与 notarization，并验证首次安装、升级、卸载后的 OBS 加载行为。

### 7.3 升级兼容

- 为设置保存版本号，插件升级时执行迁移。
- 模型资源采用独立版本号和校验；模型更新不得破坏用户既有预设。
- 对 OBS 主版本进行兼容性矩阵测试，并在发布页明确支持的 OBS 版本范围。

## 8. 质量与验收标准

### 8.1 功能验收

- 可在 Windows x64 和 Apple Silicon Mac 的 OBS 中添加、保存、重启后恢复滤镜。
- 单人和 2–4 人画面中，效果仅作用于目标人脸/皮肤，背景和衣物不应出现明显平滑或偏色。
- 人脸移动、转头、短暂遮挡时，mask 无明显闪烁、跳变或拖影。
- 关闭滤镜后输出应恢复原始画面，不留下色偏或透明度问题。

### 8.2 稳定性验收

- 切换场景、启停摄像头、热插拔设备、改变源分辨率时不崩溃。
- 模型加载失败、运行时不可用、显存/内存不足时可降级或禁用，并在日志中给出原因。
- 连续运行 2 小时不出现持续内存增长、明显帧率劣化或 OBS 无响应。

### 8.3 性能验收

- 分别在“兼容”“推荐”“高性能”基准设备上测试 720p30、1080p30 和 1080p60。
- 记录端到端帧耗时、推理耗时、GPU/CPU 占用、内存/显存、OBS 丢帧和 mask 更新频率。
- 在默认“自动”模式下，插件不应因推理而阻塞渲染线程。

## 9. 风险与约束

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 模型或依赖许可证不兼容 | 无法按预期发布 | 立项时完成代码、模型、运行时三类许可证审查。 |
| 不同摄像头色彩/曝光差异大 | 肤色算法体验不一致 | 建立多肤色、不同光照与摄像头测试集，限制极端参数。 |
| AI 推理和 OBS 编码抢资源 | 掉帧或卡顿 | 异步推理、自动降级、推荐硬编码。 |
| OBS API 或图形后端变化 | 新版本无法加载 | 基于官方插件模板，维护 OBS 版本兼容矩阵。 |
| macOS 签名/公证缺失 | 用户无法安装或加载 | 将签名、公证与干净机器验证纳入发布流水线。 |
| 滤镜链顺序不当 | 边缘伪影、效果异常 | UI 提示推荐顺序，文档列出兼容性说明。 |

## 11. 待确认需求

以下事项会影响模型选择、界面、性能目标或发布方式，需在进入 P1 前确认：

1. 首版是否必须支持 1080p60，还是 1080p30 即可？
2. Windows 最低系统版本和最低硬件基线是什么？是否允许仅 CPU 运行？
3. 是否需要适配 ARM Windows、Linux 或 Intel Mac（当前均不在范围内）？
4. 单人优先还是必须多人脸？最多支持几人？
5. 是否需要瘦脸、大眼、妆容、牙齿美白、祛痘等能力；哪些必须进入首版？
6. 是否接受开源/GPL 发布，还是必须闭源商业发布？这决定可复用代码与模型的选择。
7. 是否需要离线安装包、静默安装、自动更新、中文/英文界面？
8. 是否有可量化的质量标准，例如“背景误处理率”“人脸 mask 抖动”“最大可接受增加延迟”？
9. 是否需要品牌预设、默认强度与 UI 视觉规范？
10. 是否涉及未成年人、平台审核或肖像处理政策，需要额外提示/限制？

## 12. 参考实现方向

- 使用 OBS 官方 Plugin Template 建立跨平台插件工程。
- 使用 OBS 图形效果接口实现跨图形后端的 Shader，避免业务层直接绑定 Direct3D 或 Metal。
- 参考成熟 OBS AI 滤镜的“后台推理 + 渲染线程合成 + 异步 mask”架构；仅在许可证允许时复用代码。
- 在模型确定前先以可替换的 `Inference Engine` 接口隔离实现，避免后续更换模型或后端时重写插件主体。
